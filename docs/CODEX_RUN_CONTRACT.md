## Gate definitions (G1b/G1c/G1d)

### G1b: AMD HLS execution evidence (boardless)
Counts as CLOSED only if all are true for each required benchmark run:
1. Run executed with AMD HLS available (`--hw require`).
2. `bench_report.json` exists and `ok=true`.
3. `hardware.toolchain.available=true`.
4. `hardware.csim.ok=true` and `hardware.csynth.ok=true`.
5. HLS report artifacts exist (at least one csynth report and run logs) under the run outdir.

No simulated/mock-only run can close G1b.

### G1c: AMD Vivado implementation evidence (boardless)
Counts as CLOSED only if all are true for each required benchmark run:
1. Vivado implementation attempted by AMD Vivado in batch mode.
2. `bench_report.json` shows `hardware.vivado.implOk=true`.
3. `hardware.vivado.wns` is numeric (non-null).
4. Timing/utilization report files exist in the run outdir.
5. Evidence references target part `xc7a200t-1sbg484c`.

No stale table entry can close G1c without matching `bench_report` and report files.

### G1d: AMD power estimation evidence (boardless only)
Counts as CLOSED only if all are true:
1. Power generated by AMD Vivado `report_power` from implementation artifacts.
2. Output artifact exists with explicit estimated label (`ESTIMATED_*`).
3. Source mode documented: `vectorless` or activity-driven (`VCD`/`SAIF`).
4. No claim of measured board power is made.

G1d is estimation-only in this contract; it does NOT satisfy on-board measurement milestones.

## Estimated vs measured (strict separation)
- Estimated evidence:
  - labels must be `ESTIMATED_*`
  - acceptable for boardless confidence and G1d
  - never presented as board measurement
- Real board measurement evidence:
  - requires `build_hw/fpga_measure/power_latency_report.json`
  - requires `method="MEASURED_ON_BOARD"`
  - required for real measurement milestones (for example AMPPLUS-001)

## Non-negotiable constraints
- Do not invent results.
- Do not close G1b/G1c/G1d without real AMD-generated evidence.
- Do not present estimated artifacts as measured on board.
- Do not alter normative semantics unless a real bug is demonstrated.
- Prefer regenerating derived artifacts over manual edits.

## Canonical truth order for reconciliation
1. `bench_report.json`
2. `audit_mode_*.json` / audit outputs
3. derived tables/reports
4. older snapshots
